## Architecture Decisions

### LLM Integration
- **Contract**: Provider-agnostic OpenAI-compatible interface
- **Production model**: Qwen3 32B via vLLM
- **Local dev model**: Smaller quantized Qwen3 via Ollama
- **Configuration**: Environment-driven (model name, base URL, API key, timeout, generation params)
- **LLM role**: Language understanding, fact extraction, clarification, responses, classification, summarization
- **LLM restrictions**: No direct calculation of authenticity scores, priority scores, or rankings - those use deterministic engines
- **Evaluation**: CIP-specific multilingual eval suite (Odia, Hindi, English, code-mixed)
- **Tone & Personality**:
  - Human-like, humble, respectful, and natural conversation style
  - Avoids robotic template repetition
  - **Multilingual - replies in the detected language of the user**
  - Acknowledges concerns with empathy
  - Asks only necessary questions
  - Explains status accurately without making promises
  - Maintains professional yet approachable tone
  - **Dynamic emotional responses based on complaint type**:
    - Emergency/safety issues → Urgent, concerned, reassuring tone
    - Infrastructure problems → Empathetic, solution-focused
    - Health concerns → Compassionate, careful, action-oriented
    - Education issues → Supportive, encouraging
    - Environmental concerns → Thoughtful, community-focused
    - Administrative issues → Patient, helpful, clarifying
  - **Context-aware reactions**: Shows appropriate concern level based on severity, affected population, and vulnerability
  - **Varied phrasing**: Never repeats the same response structure; uses diverse sentence patterns, vocabulary, and emotional expressions
  - System prompt enforces: "You are a humble, helpful government assistant. Speak naturally, respectfully, and concisely. Match your emotional tone to the severity and type of concern. Show genuine care and concern. Vary your responses - never use the same phrasing twice. Acknowledge the person's feelings before addressing their issue."

### Multilingual Chat System
- **Language detection**: Automatically detect user's input language (Odia, Hindi, English, Bengali, Telugu, Tamil, etc.)
- **Response language**: Always reply in the detected language
- **Internal processing**: Translate to English for structured extraction and analysis
- **Preserve original**: Store original user text alongside translated version
- **Language fallback**: If detection confidence is low, ask in detected language or default to English
- **Code-mixed support**: Handle Hindi-English, Odia-English, and other code-mixed inputs
- **Script detection**: Support Devanagari, Odia, Latin, Bengali, Telugu, Tamil scripts
- **Translation contracts**: Use LLM-based translation for internal processing, preserve original for audit

### Geospatial Integration
- **Primary**: Google Maps Platform (Geocoding API, Places API)
- **Secondary/Fallback**: HERE Geocoding & Search API
- **Contracts**: `GeocoderProvider`, `PlacesProvider`, `MapProvider`, `GeospatialProvider`
- **Internal authority**: PostgreSQL/PostGIS with Indian government boundary datasets
- **Device GPS**: Native GPS/GNSS APIs for sender coordinates only
- **Frontend maps**: MapLibre GL JS (or benchmark-selected renderer)
- **Decision process**: Evaluate based on Indian/Odisha address accuracy, latency, coverage, licensing, cost

### Authentication & UI Architecture
- **Single login screen** for all users (citizens, MPs, officers, admins)
- **Three screens** after login based on role:
  - **User (Citizen) screen**: Chat-based AI assistant interface → submit complaints → track status → receive updates
  - **MP & Associates screen**: Dashboard with constituency issues, priority rankings, location data, complaint filer details, copilot
  - **Admin screen**: Full system management, all constituencies, user management, configuration
- **Implementation**: JWT + RBAC from the start
- **Roles**: Citizen, MP, Officer, Admin
- **Mechanism**: JWT tokens with role claims, frontend routes based on role
- **Citizen experience**: Conversational AI chat → submit complaints → track status → see "X people reported this"
- **MP experience**: Constituency-scoped dashboard → consolidated issues (one entry per unique problem) → priority list → issue details → filer info → location maps → recommendations → copilot
- **Admin experience**: Full dashboard across all constituencies → manage everything

### Duplicate Complaint Consolidation
- **Rule**: If 2+ people submit the same issue with same location/address/map pinpoint, only ONE complaint shows on MP dashboard
- **Consolidation criteria**: Category match (+0.4) + Subcategory match (+0.2) + Geographic proximity within 5km (+0.3) + Title similarity (+0.1)
- **Threshold**: Score >= 0.8 merges into existing cluster; < 0.8 creates new cluster
- **Citizen view**: Shows own submissions + cluster status + "X people reported this issue"
- **MP view**: Shows consolidated issue clusters with demand count, not individual submissions
- **Admin view**: Can drill down into individual submissions within a cluster
- **Bug fix needed**: Dead code in `consolidate_issue.py` (lines 94-99 unreachable)

### 2C: Multi-Dimensional Priority Scoring System

The priority engine evaluates issues across **16+ dimensions** to produce a comprehensive threat/impact analysis for the admin dashboard:

**Core Dimensions:**
1. **Political Impact** - Seat preservation risk, opposition attack surface, election cycle sensitivity
2. **Social Impact** - Community disruption, social cohesion, inequality effects
3. **Caste Dynamics** - Caste-based community impacts, historical sensitivities
4. **Scientific Basis** - Evidence quality, data-driven assessment, technical feasibility
5. **Natural Risk** - Natural disaster potential, weather vulnerability, geological factors
6. **Environmental Impact** - Ecological damage, pollution, sustainability concerns
7. **Innovation Potential** - Opportunity for modernization, technology adoption
8. **Societal Benefit** - Broader community value, public good contribution
9. **Humanitarian Concern** - Human suffering, dignity, basic needs fulfillment
10. **Moral/Ethical Weight** - Ethical considerations, justice, fairness
11. **Religious Sensitivity** - Religious community impacts, sacred site proximity
12. **Financial Impact** - Cost-benefit ratio, budget alignment, economic returns
13. **Futuristic/Long-term** - Strategic importance, generational impact, sustainability
14. **Educational Value** - Learning opportunities, skill development, knowledge access
15. **Historical Significance** - Heritage preservation, cultural importance
16. **Sentimental/Emotional** - Public sentiment, emotional resonance, media attention potential

**Threat Assessment:**
- **Administrative Risk** - Bureaucratic failure probability, department capacity
- **Public Criticism Risk** - Media attention, social media virality, public backlash
- **Legal/Compliance Risk** - Regulatory violations, legal exposure
- **Implementation Risk** - Execution complexity, dependency chains, timeline pressure

**Scoring Formula:**
```
Final Score = Σ(Dimension Weight × Dimension Score) × Threat Multiplier × Urgency Factor
```

**Dashboard Output:**
- Ranked priority list with score breakdown
- Threat heat map by dimension
- Risk alerts for high-threat issues
- Recommended action priority order
- Evidence traceability for each dimension

**Data Sources for Each Dimension:**
- Citizen submissions + sentiment analysis
- Demographic data
- Infrastructure data
- Historical issue patterns
- News sentiment
- Social media signals (when available)
- Government datasets
- Officer assessments
- Expert knowledge base
